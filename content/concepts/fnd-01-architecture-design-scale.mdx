# 架构与设计的尺度

“架构”不是“大图”的同义词。一个类可以承载高代价协议决定，一张系统图也可能只是在复述目录结构；判断尺度要看决定约束了谁、改变它要付出什么代价，以及需要哪些视图才能评审。

[概念目录](/concepts)提供后续主题入口，[架构思维与表达路径](/paths/architecture-thinking)把本文放在第一阶段起点。本文采用 NASA 软件架构指导、C4 抽象和 ISO/IEC/IEEE 42010:2022 的公开摘要作为边界，但不把任何一套术语宣称为普遍分类法。

## 学习问题

- 如何用影响范围和变更成本判断一个决定是否具有架构意义？
- 系统架构、应用设计与代码级设计分别需要回答什么问题？
- 架构本身、架构描述和图示为什么不能画等号？
- 何时不应把局部实现选择升级成架构决定？

## 定义与尺度边界

本文把软件架构理解为：一组跨元素、跨团队或跨生命周期约束系统演进的重要决定，以及这些决定形成的结构、关系和原则。[NASA NPR 7150.2D 第 4 章](https://nodis3.gsfc.nasa.gov/npg_img/N_PR_7150_002D_/N_PR_7150_002D__Chapter4.pdf)把软件需求、架构、详细设计和评审放在同一工程生命周期中；它面向 NASA 项目治理，不是所有组织都必须照搬的流程。

尺度不是按文件数量机械划分。系统架构关注系统与环境、主要运行单元、数据与故障边界；应用设计关注单个应用内部的模块职责与协作；代码级设计关注类、函数、算法和局部数据结构。若某个选择改变公共协议、数据所有权、安全边界或多个团队的交付方式，即使落在一个文件里，也可能是架构决定。

[ISO/IEC/IEEE 42010:2022 公开摘要](https://www.iso.org/standard/74393.html)明确区分实体的架构与表达它的架构描述，并且不规定创建描述的方法、记法、工具或记录格式。因此，本文不可用来断言“用了某种图就是有架构”，也不应把标准摘要扩张成完整标准条文。

## 核心机制

判断尺度时，依次追踪四个问题：决定作用于哪些元素，约束哪些关系，保护哪些质量目标，错误后由谁承担恢复成本。作用范围越跨边界、替换窗口越长、失败影响越难隔离，就越需要把决定提升到架构层公开评审。

[C4 Model abstractions](https://c4model.com/abstractions)用 software system、container、component 和 code 提供逐层放大的抽象。它帮助选择沟通层级，但不是通用本体：`container` 在 C4 中表示可运行或可部署单元，不等同于 Docker 容器；C4 也不替代质量属性场景、决策理由或运行验证。

下面是本站原创的尺度判断表。它回答“评审时最少要看到什么”，不是对所有组织都适用的职位或文档模板。

| 判断维度 | 系统架构 | 应用设计 | 代码级设计 |
| --- | --- | --- | --- |
| 主要问题 | 系统与环境如何分界，运行单元如何协作 | 应用内部职责如何分解与依赖 | 局部行为如何正确实现 |
| 典型影响 | 多团队、公共接口、数据或故障域 | 单个应用或相邻模块 | 单模块、类、函数 |
| 最小证据 | 上下文/容器视图、驱动力、取舍与风险 | 模块关系、接口契约、关键流程 | 类型、算法、测试与局部不变量 |
| 失效信号 | 局部改动引发跨系统事故或迁移 | 依赖循环、职责漂移、难以测试 | 错误结果、复杂度或性能热点 |

结论不是“越高层越重要”，而是让决定在恰当范围内可见。架构图若没有驱动力和边界，不能证明设计成立；代码细节若承载不可逆协议，也不可因位置低而跳过架构评审。

## 常见混淆

第一种混淆是把组织图、部署图和代码包结构当作同一结构。它们可能相关，却服务不同视点；强行一一对应会隐藏共享数据、异步控制或第三方故障边界。

第二种混淆是把“重要”与“架构级”画等号。金额计算函数可能非常重要，但若接口稳定、影响局部且可独立替换，它仍适合在代码设计层治理。反过来，一个短小配置项若改变区域故障转移或数据保留规则，就应进入架构决定。

失败模式是只保留一张总图：读者既看不见系统环境，也无法下钻责任边界，最终在事故时才发现接口和恢复所有权未定义。何时不用本文尺度表？当团队正在执行受法规、采购合同或既定架构框架约束的正式分类时，应以该制度的定义为准，再把本文四问作为补充检查，不应替换合规术语。

## 说明性场景

一个报销系统计划把“付款任务”从 Web 进程移到独立执行器。若只看代码，这是移动一个调用；沿四问检查后会发现它新增异步协议、任务状态、重试副作用和独立部署故障域，因此应在系统或应用架构层记录。

团队用 C4 Context 保持员工、报销系统和银行的外部关系，用 Container 层显示 Web、API、数据库与任务执行器，再用代码设计处理任务序列化。失败边界保持可见：图不能证明重试安全，仍需单独定义幂等与核对机制。这个场景为原创推演，不代表 NASA、C4 或 ISO 给出的标准方案。
