# 模块边界与应用架构

## 为什么学

多数系统不是先被流量压垮，而是先被变化成本拖慢。模块边界决定哪些代码一起变化、哪些知识可以隐藏，也决定未来拆分服务时能否保留清楚的所有权。

## 前置能力与跳过条件

读者应能从上下文图和 ADR 说明系统约束。如果已经能为熟悉应用画出模块图，标明公开接口、数据所有者和依赖方向，并能用变化原因而非技术栈名称解释为何保持单体或拆分服务，可以跳过必读起点。

## 核心问题

- 内聚、耦合、依赖方向和信息隐藏如何影响变化成本？
- 分层、Hexagonal、Clean、Vertical Slice 和模块化单体分别在约束什么？
- DDD 的限界上下文如何帮助划分责任，而不是增加一套名词？
- 什么证据足以支持拆分服务，什么情况应继续保持模块化单体？

## 推荐学习顺序

1. 按变化原因和业务责任识别候选模块。
2. 标出公开接口、数据所有者、依赖方向和跨模块写入。
3. 用 Hexagonal、Clean 或 Vertical Slice 检查依赖是否真正指向稳定边界。
4. 依据独立交付、扩缩容与故障隔离证据判断是否拆分服务。

## 必读起点

[Common web application architectures](https://learn.microsoft.com/en-us/dotnet/architecture/modern-web-apps-azure/common-web-application-architectures) 与 [Architecture Patterns with Python](https://www.cosmicpython.com/book/preface.html) 用来建立边界、依赖方向和部署单元的区别。

## 查漏补缺

按当前项目缺口选读 [Architectural Design Principles](https://github.com/mehdihadeli/awesome-software-architecture#architectural-design-principles)、[Modular Monolith](https://github.com/mehdihadeli/awesome-software-architecture#modular-monolith)、[Domain Driven Design](https://github.com/mehdihadeli/awesome-software-architecture#domain-driven-design)、[Hexagonal Architecture](https://github.com/mehdihadeli/awesome-software-architecture#hexagonal-architecture)、[Clean Architecture](https://github.com/mehdihadeli/awesome-software-architecture#clean-architecture) 与 [Vertical Slice Architecture](https://github.com/mehdihadeli/awesome-software-architecture#vertical-slice-architecture)。这些入口属于社区索引，应继续回到原作者、官方文档或可检查的实现核对架构事实。

## 深入拓展

从上述专题进入模式原作者、示例项目和重构材料，用具体代码核对依赖方向是否真的成立。模式名称不能替代对编译依赖、运行时调用和数据所有权的检查。

## 用本站案例深化

- [Micro Frontends + single-spa](/cases/micro-frontends-single-spa)：检查按业务切片、独立交付与共享运行时之间的张力。
- [OpenAI Agents SDK](/cases/openai-agents-sdk)：比较 Manager、Handoff 与确定性代码编排，观察控制权也是一种职责边界。

## 实践产出

保留一张标明所有权和依赖方向的模块图。每个模块还应标出公开接口与数据所有者，使边界可以被代码和团队责任共同核对。

## 检查点

为一个熟悉应用画出模块图，标明公开接口、数据所有者和依赖方向，并解释为什么当前应该或不应该拆成独立服务。能够用变化原因而不是技术栈名称说明边界，即可进入下一阶段。
