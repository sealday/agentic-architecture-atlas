# Agent 平台与模型网关

## 为什么学

模型网关处理的不只是统一 API，Agent 平台处理的也不只是框架托管。模型与能力路由、租户身份、预算、Guardrail、长时状态、协议、评估和隔离分布在不同控制点；如果不先划清责任，单个“平台层”会掩盖失败与越权路径。

## 前置能力与跳过条件

读者应先理解 Agentic 控制权、生产隔离和恢复责任。如果已经能为一次 Agent 调用链说明路由依据、调用者身份、虚拟 Key、配额与成本归属、Guardrail 时序、状态所有者、A2A/MCP 边界、评估信号和执行隔离，可以跳过必读起点。

## 核心问题

- 模型、供应商和工具能力由谁路由，回退如何避免改变语义或合规边界？
- 用户、服务、Agent、虚拟 Key 和上游凭据如何关联，配额与成本记到谁名下？
- 输入、输出、工具调用和人工审批的 Guardrail 应分别放在哪个强制点？
- 编排状态、业务事实、长期记忆、A2A 任务与 MCP 工具会话如何分界？
- 质量、延迟、成本、安全事件和租户故障半径如何被观察和隔离？

## 推荐学习顺序

1. 先定义模型与能力目录、路由约束、回退条件和不可跨越的合规边界。
2. 再串起调用者身份、虚拟 Key、上游凭据、配额、预算和成本归属。
3. 然后标出输入、输出、工具执行前后与人工审批位置，明确每道 Guardrail 能阻止什么。
4. 接着划分编排状态、业务事实与长期记忆，再判断 A2A 或 MCP 是否对应真实的跨 Agent 或工具接入边界。
5. 最后建立评估、追踪、成本和安全信号，并用租户、任务和执行环境隔离故障。

## 必读起点

[LiteLLM Virtual Keys](https://docs.litellm.ai/docs/proxy/virtual_keys) 用于理解虚拟 Key、预算与网关治理入口；[OpenAI Agents SDK 多智能体文档](https://openai.github.io/openai-agents-python/multi_agent/) 用于理解 Manager 与 Handoff 等控制原语；[A2A 文档](https://a2a-protocol.org/latest/) 与 [MCP 入门](https://modelcontextprotocol.io/docs/getting-started/intro) 用于区分跨 Agent 任务互操作和工具或数据接入。它们描述各自机制，不共同构成一套开箱即用的平台架构。

## 查漏补缺

[Envoy AI Gateway 入门](https://aigateway.envoyproxy.io/docs/getting-started/) 可用于比较基于网关基础设施的模型流量入口，[Awesome Agentic Engineering Resources](https://github.com/EthicalML/awesome-agentic-engineering-resources) 可用于发现评估、安全、编排和平台材料。后者是社区索引；二者都不替代对目标版本、供应商契约和本地风险的核验。

## 深入拓展

只有在真实平台需求出现后，才沿官方文档和上游仓库核对路由 API、身份传播、状态存储、协议版本、追踪字段和失败恢复。不要从案例或索引推导供应商等价性，也不要把协议互通写成权限、事务或结果质量保证。

## 实践产出

为一次端到端 Agent 请求保留一张平台控制表：记录路由依据、身份和凭据、配额与成本、每道 Guardrail、状态所有者、协议边界、评估与追踪信号，以及租户和执行环境隔离。

## 检查点

追踪一次带工具副作用的 Agent 请求，说明网关和编排器各自决定什么、身份和预算如何传播、Guardrail 在何处强制、状态在哪里恢复、A2A/MCP 是否必要，以及哪项本地证据仍不足以支持生产结论。
