# 架构权衡分析方法（ATAM）

ATAM 面向已有候选架构：它把关键质量属性场景映射到架构方法，寻找参数变化会改变结果的敏感点、同时影响多个属性的权衡点，以及需要处置的风险。[方法目录](/methods)提供方法入口，[架构思维与表达路径](/paths/architecture-thinking)给出学习位置。

## 学习问题

- 质量属性场景如何组织成可评估的 utility tree？
- 如何从架构方法中识别敏感点与权衡点，而不是只列优缺点？
- 风险、非风险与风险主题各需要什么证据？
- ATAM 为何既不是架构设计方法，也不是合规认证？

## 输入与参与者

输入包括业务驱动力、按[质量属性场景写法](/quality-attributes/qa-01)表达并排序的场景、候选架构视图、关键架构方法、约束和已有验证证据。业务代表说明价值与失败影响，架构师解释方法和参数，开发、运维、安全及依赖方挑战假设，评估主持人维护证据链。

QA-01 目前有尚未发布的 QA-00 前置缺口，本批次不关闭 QA-00。评估可使用 QA-01 已发布的场景格式，但结论必须保留这一学习图谱缺口，不能把它误写为方法输入已经完整。

## 步骤

1. 对齐业务驱动力、评估范围与不可更改约束。
2. 将质量目标展开成 utility tree，以属性、关注点和具体场景分层，并按业务重要性与技术难度排序。
3. 架构师说明实现高优先级场景的架构方法、关键参数和依赖假设。
4. 逐场景追问参数变化：显著改变单个质量结果的是敏感点，同时改善一项却损害另一项的是权衡点。
5. 依据可验证证据记录风险与非风险，再聚合重复原因形成风险主题和后续验证计划。

本站原创的证据判定表如下：

| 观察 | 判定 | 最小证据 | 后续动作 |
| --- | --- | --- | --- |
| 单元规模增大使恢复时间越过阈值 | 敏感点 | 参数、场景度量、边界值 | 演练临界区间 |
| 更小单元降低故障半径但增加运维成本 | 权衡点 | 至少两个属性的方向变化 | 明确授权取舍 |
| 跨单元依赖使隔离失效且尚无演练 | 风险 | 失败路径与证据缺口 | 指定缓解与验证 |
| 隔离演练已满足目标且适用前提明确 | 非风险 | 有日期的结果与适用范围 | 保留复核条件 |

## 产物

产物包括排序后的 utility tree、场景到架构方法的映射、敏感点、权衡点、风险、非风险、风险主题、未决问题和验证计划。每一项应链接具体场景与证据；“大家认为可行”不是非风险证据，“使用某云服务”也不是风险已消除的证明。

## 完成判断

当高优先级场景均映射到可说明的架构方法，关键参数及假设可见，风险与非风险有证据边界，重复风险已归纳为主题，并为未验证项指定责任人和截止条件时，评估才完成。完成意味着风险被揭示并可管理，不意味着架构获得永久批准。

## 常见失败

常见失败是没有具体场景就讨论通用优缺点，或把技术名称当作架构方法；另一个失败是只登记风险而不登记已证实的非风险，使后续团队无法区分未知与已验证边界。若架构师在会上临时设计全部结构，评估会失去稳定对象。

ATAM 不是架构设计方法，不负责从空白产生架构；它也不是认证、审计签字或形式化正确性证明。没有候选架构、没有可排序场景，或决策只是局部可逆实现时，不应使用完整 ATAM；应先采用[质量属性工作坊](/methods/mth-01)、架构设计活动或轻量代码评审。

## 与其他方法的衔接

MTH-01 先发现并排序场景，[权衡、敏感点与风险](/concepts/fnd-04)提供判断词汇，MTH-02 再对已有架构形成评估结果。被接受的取舍可进入 ADR，风险处置则进入演练、监控或重设计，新的证据反过来更新复核。

在 [AWS Cell Architecture + Shuffle Sharding 案例](/cases/aws-cell-shuffle-sharding)中，评估对象不是“AWS 是否可靠”，而是 cell 大小、租户映射、跨 cell 依赖和 shuffle 宽度怎样改变故障半径、容量效率、恢复时间与运维复杂度。

## 完整演练

订单平台已有 cell 架构候选。utility tree 的最高优先场景要求：任一 cell 失效时，受影响租户不超过总量的 5%，未受影响租户的成功率保持 99.9%；另一场景要求扩容操作在十五分钟内完成。

团队把 cell 租户数识别为敏感点：从 4% 增至 8% 会直接越过故障半径阈值。减小 cell 同时提高隔离性并增加基础设施与值班成本，因此构成权衡点。评估又发现身份服务是跨 cell 同步依赖，尚无隔离演练，记录为风险；已有混沌演练证明单 cell 数据面失效不会扩散，且版本与流量前提明确，记录为非风险。

最终风险主题是“控制面共享依赖削弱数据面隔离”。团队决定先拆除身份查询的同步运行时依赖，再用同一场景复测；评估没有替团队设计完整新架构，也没有签发认证。

## 来源

utility tree、架构方法、敏感点、权衡点、风险与非风险的评估语境依据 [SEI《The Architecture Tradeoff Analysis Method》](https://www.sei.cmu.edu/library/the-architecture-tradeoff-analysis-method/)；架构驱动力、描述与验证证据的可追溯关系用 [NASA Software Engineering Handbook 7.07](https://swehb.nasa.gov/spaces/SWEHBVD/pages/102695632/7.07%2B-%2BSoftware%2BArchitecture%2BDescription)交叉核对。证据判定表与演练为本站原创归纳。
