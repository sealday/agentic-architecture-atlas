# 架构案例双层写作与项目级 Skill 设计规格

**日期：** 2026-07-23  
**状态：** 待用户审阅  
**方案：** B——双层阅读结构  
**仓库：** `sealday/agentic-architecture-atlas`  
**本地根目录：** `/Users/seal/projects/tego-arch`

## 1. 背景

站点现有十五篇架构案例，已经具备较完整的事实边界、源码锚点、权衡和迁移分析。问题不是信息不足，而是太多段落同时承担机制解释、版本说明、例外、风险和来源证明。读者需要持续以“审阅材料”的方式阅读，难以先建立整体理解，再按需深入。

本次采用双层阅读结构：

- **主叙事层负责读懂。** 用具体冲突、任务流、微型场景和短段落解释系统为何这样设计。
- **证据层负责核验。** 保存版本、固定 commit、源码位置、术语边界、例外和来源链接，供读者按需展开。

这不是降低技术深度，也不是删除证据，而是重新安排阅读顺序。

## 2. 目标

- 在项目目录内创建一个可持续调整的写作 Skill，供新增、改写和审阅架构案例时复用。
- 按该 Skill 改写现有十五篇案例，明显降低连续高密度阅读的疲劳。
- 保留所有已经核验的事实、来源、版本锚点、架构结论和迁移边界。
- 保留现有案例 URL、front matter、十个 H2 和迁移分析 H3 契约。
- 让正文既适合第一次接触该架构的工程师顺序阅读，也适合资深读者快速定位证据。
- 将“可读性”纳入可重复检查，而不是依赖一次性的人工感觉。

## 3. 目标读者

默认读者是：

> 有软件工程经验，但第一次系统接触当前案例所分析架构的工程师。

文章可以使用必要的工程术语，但首次出现时必须给出上下文。熟悉上游项目的读者不应被冗长入门解释拖慢；不熟悉项目的读者也不应被连续的组件名和源码名词挡在门外。

## 4. 非目标

本次不做：

- 删除源码导读、来源、版本或证据标签。
- 把案例改成营销文章、项目介绍或“最佳实践”清单。
- 编造用户故事、事故、性能数字、生产经验或第一人称经历。
- 为追求轻松感加入与论点无关的类比、段子和重复总结。
- 改变案例分类、目录顺序、路由和现有研究结论。
- 新增服务端、CMS 或第三方写作依赖。
- 以硬性字数上限替代编辑判断。

## 5. 项目级 Skill

### 5.1 名称与位置

Skill 名称固定为 `writing-architecture-cases`，创建在：

```text
.codex/
└── skills/
    └── writing-architecture-cases/
        ├── SKILL.md
        ├── agents/
        │   └── openai.yaml
        └── references/
            ├── article-contract.md
            └── review-checklist.md
```

`.codex/skills` 使 Skill 与仓库一起版本化，后续调整写作风格时不依赖个人机器上的全局配置。

`SKILL.md` 只保留触发条件、核心原则和执行流程。详细章节契约与审阅清单放入 `references/`，按需读取，避免 Skill 本身过长。

### 5.2 触发范围

在以下任务中使用该 Skill：

- 新增架构案例；
- 重写或扩写 `content/cases/*.mdx`；
- 审阅案例的叙事、密度、证据边界或 AI 腔；
- 将博客、论文、官方文档和源码整理成案例；
- 在不损失技术证据的前提下提高可读性。

纯粹修改 front matter、修正单个链接或执行机械格式化时不强制触发。

### 5.3 Skill 的来源融合

Skill 吸收以下原则，但形成自己的项目契约：

- ECC article-writing：具体先行、例子先于抽象、用证据替代形容词、删除 AI 填充语。
- Sentry 写作指南：从问题或结论进入、每段一个问题、显式说明权衡与限制。
- Google 技术写作：每段一个主题，关键信息前置，在对比或转折处换段。
- Microsoft 可扫描内容：短标题、短段落、首屏尽快给出价值。
- Diátaxis：解释层与参考层分开，不让证据目录吞没叙事。
- sciwrite：强动词、术语一致、句子结构清楚、引用完整。
- avoid-ai-writing：只检测套路，不用模板化“去 AI”改写覆盖作者判断。

不复制任何作者的个人声线，也不把 ECC 的高压缩“operator voice”原样套用到长篇案例。

## 6. 双层文章契约

### 6.1 主叙事层

主叙事层按“冲突—机制—结果—边界”推进。

每篇文章开头在三个自然段内完成：

1. 给出一个具体冲突、失败模式或反直觉结论；
2. 说明该案例提供了什么可迁移的观察角度；
3. 交代分析所基于的版本与证据范围，详细锚点进入证据层。

每个主要章节只承担一个主问题。一个段落只推进一个中心意思，通常为 2–4 句；出现对比、例外、层级切换或新的控制权主体时换段。

主叙事优先回答：

- 谁拥有控制权？
- 状态存在哪里？
- 一次任务如何流动？
- 失败后由谁恢复？
- 这个设计换来了什么，又牺牲了什么？

### 6.2 证据层

证据层使用原生 MDX/HTML `details` 与 `summary`，不新增运行时依赖。统一呈现为可展开的“证据卡”。

适合进入证据卡的内容：

- 固定版本、tag 和 commit；
- 具体源码文件、类型、函数和调用链；
- 术语的版本差异；
- 支持同一结论的多条来源；
- 不影响首次理解的实现例外；
- 对推断范围的补充说明。

必须留在主叙事中的内容：

- 改变结论的关键限制；
- 安全、权限、资金或不可逆副作用风险；
- 读者若忽略就会误用的边界；
- 理解任务流必需的组件或状态。

证据卡只补充正文，不负责首次提出关键概念。折叠后，文章仍应能形成完整、无误导的论证。

### 6.3 信息密度节奏

以下规则是编辑目标，不作为绝对字数禁令：

- 每段一个中心意思，通常 2–4 句，最多不超过 6 句。
- 单句超过约 80 个中文字符时优先拆分；技术标识符或链接导致的长度除外。
- 单段超过约 200 个中文字符时必须检查是否混入第二个主题。
- 一个普通叙事段落尽量只引入不超过 3 个新标识符。
- 连续两个高密度段落后，增加一个短解释、微型场景、检查点或图表导读。
- 表格和代码块前后各有一句解释，说明“为什么看”和“看完得到什么”。
- 删除重复结论、同义排比、虚假悬念和无信息量的过渡句。

“低密度内容”必须完成至少一个功能：建立场景、解释术语、承接转折、复述任务流、提出检查问题或给读者一个停顿点。纯填充不计入。

### 6.4 微型场景

每篇案例至少包含一个可追踪的微型场景，最好放在“控制权与任务流”或“生产化分析”中。

场景应满足：

- 只使用已证实机制，或明确标记为“说明性场景”；
- 描述一个请求、状态变化、失败或恢复过程；
- 不编造真实客户、事故、数字或上游项目承诺；
- 最终回到架构判断，而不是成为孤立故事。

示例形态：

> 一个任务已完成付款，但发送确认邮件时执行器崩溃。重启后，系统如何知道哪一步可以重试，哪一步必须补偿？

随后用案例中的实际机制回答。

## 7. 现有十章结构的改写规则

现有十个 H2 保持顺序和标题不变。

### `## 学习问题`

保留 4–6 个真正驱动阅读的问题。删除答案高度重叠的问题。问题优先围绕控制权、状态、失败、边界和迁移。

### `## 一页摘要`

先用两到四个短段落建立全貌，再使用一张紧凑表格。不要用连续证据标签打断摘要。

### `## 事实边界`

正文只保留会影响结论的版本边界。commit、文件路径和来源拓扑进入证据卡。

### `## 架构图`

图前说明要观察的关系，图后按一次真实任务流解释。避免逐节点复读图中的标签。

### `## 控制权与任务流`

使用微型场景串联角色、状态和失败路径。明确控制权何时转移、谁可以终止或重试。

### `## 关键源码导读`

先给“阅读目的”，再给最短的源码路径。详细符号清单和辅助锚点进入证据卡。

### `## 架构决策与权衡`

每个决策采用“解决的问题—采用的机制—代价—适用边界”四部分。避免只列优点和缺点。

### `## 生产化分析`

围绕具体故障或运维动作展开。关键风险不得折叠。监控项和延伸证据可以进入证据卡。

### `## 可迁移经验`

保留三个既有 H3：

- `### 可直接复用的机制`
- `### 只能有限类比的部分`
- `### 不应照搬的部分`

每一项都说明迁移条件，不写脱离上下文的口号。

### `## 来源`

保留完整、可核验的来源。主来源、源码来源和补充解释分组，避免无层次链接堆叠。

## 8. 证据标签

继续区分：

- **已证实事实**
- **基于证据的推断**
- **个人分析**

但不再要求每个自然段都重复标签。标签可以作用于一个短小、主题一致的内容块；块内一旦从事实转向推断，必须换块或重新标记。

证据卡不能把推断伪装成事实。引用支持的是机制而非最终判断时，应明确写出“由此推断”。

## 9. 写作工作流

Skill 固定使用以下流程：

1. **确定读者与学习问题。** 写出本篇读者需要带走的 3–5 个判断。
2. **建立证据清单。** 区分官方文档、源码、论文、工程博客和个人分析。
3. **设计主叙事。** 每节只解决一个问题，先排任务流，再安排术语。
4. **撰写主叙事层。** 先不堆源码锚点，确保折叠证据后仍能读懂。
5. **附加证据层。** 把版本、路径、例外和多来源证明放入证据卡。
6. **执行密度检查。** 查找长句、长段、标识符堆积和连续高密度区。
7. **执行 AI 腔检测。** 只标记套路化开头、空洞排比、口号式结尾和假悬念，再人工改写。
8. **执行事实与结构校验。** 检查标签、来源、版本、十个 H2、迁移 H3 和 front matter。
9. **浏览器阅读检查。** 同时检查桌面和移动宽度下的首屏、段落节奏、证据卡和表格。

## 10. 可读性检查器

在 Skill 中附带一个只报告、不阻断构建的检查脚本，扫描 `content/cases/*.mdx` 并输出：

- 超过建议长度的句子；
- 超过建议长度的正文段落；
- 连续高密度段落；
- 可能过度重复的证据标签；
- 章节是否缺少场景提示；
- 证据卡是否为空或只包含结论。

首版检查器只作为编辑反馈，不加入 `npm run verify` 的失败条件。现有内容验证器继续负责确定性的结构和元数据契约。待十五篇全部改写并积累误报样本后，再决定哪些规则适合升级为 CI 门禁。

## 11. Skill 的测试策略

创建 Skill 前先记录无 Skill 的基线行为，至少覆盖三种任务：

1. 将一段证据密集的源码分析改成更易读正文；
2. 从一组来源起草一个案例章节；
3. 审阅一篇含有 AI 腔和虚假低密度内容的段落。

基线重点观察：

- 是否直接删掉事实来换取短小；
- 是否把证据全部塞回主叙事；
- 是否编造故事或经验；
- 是否只做句子润色，没有重新安排信息层级；
- 是否把“低密度”理解为重复和填充。

完成 Skill 后用同类新样本复测。验收标准是：保持事实和边界，正确形成主叙事与证据层，并能指出不可折叠的关键风险。

## 12. 十五篇文章的实施顺序

实施分为三步，但不中断用户已要求的完整交付：

1. **校准样本：** 先改写一篇高密度案例 `temporal-saga-durable-execution.mdx`，再改写一篇已有较好具体示例的 `ros2-dds-agent-lifecycle.mdx`。两篇用于校准 Skill，不作为等待用户确认的中间门。
2. **批量改写：** 按目录顺序处理其余十三篇，保持每篇事实、来源和迁移结论。
3. **整体编辑：** 横向检查术语、证据卡命名、开头重复度、场景多样性和段落节奏，避免十五篇变成同一模板。

批量改写允许每篇保留自己的叙事重心。Skill 定义质量契约，不规定所有文章使用相同句式。

## 13. 验收标准

### 内容完整性

- 十五篇案例全部完成双层结构改写。
- 已有事实、来源、固定版本和关键边界没有无依据丢失或改变。
- 所有案例保留 front matter、稳定 slug、十个 H2 和规定的迁移 H3。
- 每篇至少有一个合规的微型场景或反例。
- 折叠全部证据卡后，主叙事仍能独立说明架构和主要权衡。

### Skill 完整性

- Skill 位于 `.codex/skills/writing-architecture-cases/`。
- 通过 Skill 目录格式校验。
- 具备有效的 `agents/openai.yaml`。
- 基线测试、复测和误区修正有可重复的验证记录。
- 可读性检查器可以扫描单篇或全部案例，并以退出码 0 稳定完成报告。

### 工程验证

- `npm run test`
- `npm run validate:content`
- `npm run check:catalog`
- `npm run typecheck`
- `npm run build`
- `npm run verify`
- 桌面与移动宽度浏览器抽查
- Git 工作区无意外文件，改动提交并推送至 `main`

## 14. 风险与控制

### 事实被“润色”掉

改写前后逐节对照关键事实、来源和边界。删除只能针对重复表达，不能针对证据本身。

### 证据卡成为信息垃圾桶

每张卡只有一个主题，标题说明展开后能得到什么。改变结论的限制留在正文。

### 所有文章模板化

统一的是阅读契约，不是开头句式。横向审阅重复的钩子、过渡和结尾，保留每个案例自己的核心冲突。

### 低密度变成注水

任何新增内容都必须承担场景、解释、过渡、检查或总结功能。无法说明作用的句子删除。

### 自动检查误报

首版只报告不阻断。规则根据十五篇真实文章的结果再校准。

## 15. 交付物

- 项目级 `writing-architecture-cases` Skill；
- Skill 的参考契约、审阅清单和可读性检查器；
- 按 Skill 改写后的十五篇案例；
- 必要的证据卡样式与测试；
- 完整构建和浏览器验证记录；
- 提交并推送到远程 `main` 的最终变更。
