# 多文章软件架构学习路线设计

## 目标

把当前 `/paths` 的单篇长路线重构为一套可逐篇学习、可持续扩写的内容体系，面向“有开发经验、第一次系统学习架构的工程师”。

路线需要同时解决四个问题：

1. 读者能从总览判断自己应该从哪里开始。
2. 每个阶段有独立文章，能够承载资源、案例、练习和检查点。
3. 四条专题分支即使案例尚未完全覆盖，也有可阅读、可继续扩写的稳定入口。
4. 学习路线引用的所有外部来源都统一登记在资料库，读者可以区分官方事实来源与社区导航来源。

## 信息架构

学习路径由 11 篇文章组成。

### 总览

- `/paths`

总览说明适用读者、使用方式、跳学规则、六阶段主干、四条专题分支，以及如何把阅读转换为架构产出。

### 六篇主干文章

1. `/paths/architecture-thinking`：架构思维与表达
2. `/paths/module-boundaries`：模块边界与应用架构
3. `/paths/distributed-systems`：分布式系统基础
4. `/paths/reliability-state`：可靠性与状态管理
5. `/paths/production-governance`：扩展、隔离与生产治理
6. `/paths/agentic-architecture`：Agentic 架构专项

### 四篇专题文章

1. `/paths/cloud-native-platform`：云原生与平台
2. `/paths/collaborative-state-frontend`：协作状态与前端架构
3. `/paths/edge-physical-agents`：边缘与物理智能体
4. `/paths/agent-platform-gateway`：Agent 平台与模型网关

四篇专题文章首版采用“可阅读骨架”，而不是空白占位符。每篇必须说明学习目标、知识地图、推荐顺序、精选来源、已有站内案例、当前覆盖范围和后续待补主题。后续直接在稳定 URL 上扩写。

## 总览页设计

总览页承担导航和决策，不重复六篇主干文章的完整正文。

页面依次包含：

1. 适用读者与使用方法。
2. 一张 ImageGen 生成的横向视觉路线图。
3. 一张 Mermaid 精确路线图。
4. 六阶段摘要与入口。
5. 四条专题分支摘要与入口。
6. 阶段产出清单和跳学规则。

### ImageGen 视觉路线图

图片用于建立整体印象，表现读者从局部代码视角逐步走向模块、分布式系统、可靠性、生产治理和 Agentic 系统，再从主干进入四个专题方向。

图片采用宽幅编辑插画或信息图风格，控制文字数量。阶段名称、依赖方向和跳转关系由页面正文与 Mermaid 承担，避免生成图片中的文字失真影响信息准确性。

最终图片必须复制到仓库的 `static/img/paths/` 下，使用稳定、语义化文件名，并由总览页直接引用。不能只保留在 ImageGen 默认输出目录。

### Mermaid 路线图

Mermaid 提供准确、可维护的结构表达：

- 六个主干阶段按依赖关系串联。
- 第五或第六阶段之后可进入相关专题分支。
- 已掌握检查点的读者可以跳过对应阶段。
- 专题分支不是主干的必修延长线。

Mermaid 不重复图片的氛围表达，而是承担节点名称、关系和阅读方向。

## 单篇文章契约

六篇主干文章统一采用以下结构：

1. 为什么学
2. 前置能力与跳过条件
3. 核心问题
4. 推荐学习顺序
5. 必读起点
6. 查漏补缺
7. 深入拓展
8. 用本站案例深化
9. 实践产出
10. 检查点
11. 上一篇 / 下一篇

每篇文章围绕一个阶段问题组织，不把资料库复制成链接清单。外部来源负责建立通用概念，站内案例负责验证机制、边界和取舍，实践产出负责把阅读转化为可复查能力。

专题文章沿用上述结构，但增加两个明确章节：

- 当前已覆盖
- 后续待补

“后续待补”使用主题清单描述内容缺口，不能出现无上下文的空白待办标记、空章节或尚未支持的结论。

## 资料库设计

`/references` 从首发案例来源页升级为全站统一来源目录。学习路径出现的每个外部链接都必须能在资料库找到对应登记。

来源分为五类：

1. 官方文档
2. 官方仓库
3. 论文与工程演讲
4. GitHub 索引
5. 工程博客与第三方教程

每个来源条目至少记录：

- 名称
- URL
- 来源类型
- 适用阶段或专题
- 在路线中的用途
- 核查日期
- 使用边界

### 来源等级

- **必读起点**：优先使用官方文档、官方仓库、公认教材或奠基性论文。
- **查漏补缺**：可以使用高质量 GitHub 索引、工程博客和第三方教程。
- **深入拓展**：可以使用专题索引、论文、演讲、案例复盘和更窄的官方资料。

第三方博客和社区索引不能作为核心架构事实的唯一证据，也不进入主干的“必读起点”。它们用于发现知识缺口、提供解释视角或引导到更具体的一手资料。

资料库首轮应覆盖当前路线已经使用的全部来源，并为四个专题补充经过筛选的入口，例如：

- Awesome Software Architecture
- System Design Primer
- roadmap.sh Software Architect
- Awesome Front-end System Design
- CNCF Curriculum
- Kubernetes、Yjs、KubeEdge、ROS 2 官方资料
- OpenAI Agents SDK、LangGraph、Google ADK、A2A、MCP 官方资料
- LiteLLM 与 Envoy AI Gateway 官方资料
- 适合 Agentic Engineering 的社区索引

## 导航与兼容性

- 保留 `/paths`，不破坏现有发布入口。
- 学习路径侧栏展示总览、六篇主干和四篇专题。
- 主干文章提供线性的上一篇 / 下一篇。
- 第六阶段的下一步指向专题选择，而不是强制进入某一专题。
- 专题文章返回总览，并链接与本专题相关的主干阶段。
- 不创建重定向，因为现有阶段内容原本没有独立 URL。

## 内容迁移

当前 `/paths` 中的六阶段正文迁移到对应独立文章。迁移时保留：

- 学习动机
- 核心问题
- 资源等级
- 案例映射
- 检查点
- 下一步关系

总览只保留摘要和入口，避免与独立文章形成两份需要同步维护的完整正文。

四条现有专题分支扩写为独立的可阅读骨架。它们引用资料库中的来源，不在首版制造未经证据支持的教程细节。

## 测试与验证

### 自动化契约

学习路径测试应验证：

- 11 篇文章和稳定 slug 全部存在。
- 总览包含 ImageGen 图片引用和 Mermaid 代码块。
- 六篇主干具备统一章节。
- 四篇专题具备统一章节、当前覆盖与后续待补。
- 主干上一篇 / 下一篇关系连续。
- 每个外部链接都在资料库登记。
- 第三方来源不被标为主干必读起点。
- 当前全部 canonical case 至少被一篇学习路径文章引用。
- 图片资产实际存在于仓库。

内容验证、类型检查和 Docusaurus 构建继续作为发布门槛。

### 视觉验证

在桌面和移动宽度检查：

- 总览图片是否清晰、不过度抢占首屏。
- Mermaid 是否可读且没有横向页面溢出。
- 11 篇文章的侧栏顺序是否符合主干和专题结构。
- 上一篇 / 下一篇导航是否明显。
- 图片暗色模式下是否仍有足够边界与对比。

## 非目标

本次不做以下工作：

- 不一次性把四篇专题扩写成完整课程。
- 不新增交互式进度追踪、账号或学习状态。
- 不为每个阶段分别生成图片。
- 不把资料库做成新的前端检索应用。
- 不重写现有案例正文。

## 完成标准

当以下条件同时满足时，本次改造完成：

1. `/paths` 成为清晰的总览页。
2. 六篇主干和四篇专题均可独立阅读并通过侧栏访问。
3. ImageGen 路线图与 Mermaid 路线图同时出现在总览。
4. 资料库登记学习路线使用的全部外部来源，并明确来源类型与用途。
5. 自动化测试、内容验证、类型检查和生产构建通过。
6. 桌面与移动视觉检查没有阻断性问题。
