# Agent 网关架构案例设计

## 背景与批准状态

用户希望把三个公开、可核验、侧重点互补的 Agent 网关案例写入案例库。此前已先行完成互联网检索，并向用户分别说明 New API、LiteLLM Proxy 与 Kong AI Gateway 的案例主线；用户确认采用这三个案例并要求按项目内 `writing-architecture-cases` skill 写入。

## 目标

新增三篇完整架构案例，使读者能够分别回答：

1. 多模型、多渠道与多 Key 怎样形成可检索、可隔离的调度池？
2. 真实上游密钥怎样转换成可撤销、可限额、可计费的租户凭据？
3. 模型路由、重试、故障转移与熔断怎样组合成有边界的恢复流程？

三篇文章必须独立成立，不能写成重复的“网关功能列表”。合并阅读时，它们应覆盖上游资源调度、调用者治理和请求级高可用三个层次。

## 案例设计

### New API：渠道池与多 Key 稳定性

- **核心冲突：** 同一模型背后有不同厂商、Base URL、账号和 Key；网关既要快速选路，也不能继续把故障凭据暴露给请求。
- **主流程：** 请求分组与模型匹配 → Ability/缓存候选 → 最高优先级层 → 层内权重选择 → Key 轮询 → 失败重试、跳过或渠道禁用。
- **读者结论：** 优先级表达故障转移层次，权重表达同层流量分配，两者不能混写；健康恢复只能是合规隔离与重新启用，不能用于规避上游平台风控。
- **证据边界：** 固定公开 tag/commit 与官方文档；不编造生产流量、账号成功率或封禁恢复指标。

### LiteLLM Proxy：虚拟密钥与多租户治理

- **核心冲突：** 共享真实 Provider Key 会放大泄露、越权、盗刷和责任归属风险。
- **主流程：** 虚拟密钥鉴权 → key/team 模型权限交集 → 多级预算、RPM/TPM/并发检查 → Guardrail → 代持上游 Key 调用 → 费用归属与管理审计。
- **读者结论：** 统一 API 的安全价值不在隐藏供应商，而在建立可撤销、可限权、可计费、可追责的凭据边界。
- **证据边界：** 固定版本至少高于安全公告已修复版本；明确 OSS 与 Enterprise 能力；管理审计不等于完整推理内容审计，预算也不等于强一致预授权账本。

### Kong AI Gateway：路由、重试与熔断

- **核心冲突：** “智能选模型”同时面对会话粘性、延迟、成本、容量与故障恢复，单一算法不能解决所有目标。
- **主流程：** provider-agnostic API → Upstream/target 候选 → 选定负载均衡策略 → failure criteria 判定 → 同目标或异目标重试 → 失败阈值熔断 → 冷却后试探恢复。
- **读者结论：** 选路、重试和熔断必须共享明确的失败分类与终止预算；Agent 工具调用若已经产生外部副作用，不能套用模型推理的透明重试。
- **证据边界：** 明确 Kong Gateway 3.10+ 与 3.13+ 的功能边界，并标注 Advanced/商业能力；官方配置文档证明机制，不证明生产 SLO 或成本降幅。

## 统一文章合同

每篇文章都必须：

- 保留十个固定 H2 与三个固定“可迁移经验”H3；
- 在开头三段内给出具体冲突、可迁移视角与证据范围；
- 使用 `已证实事实`、`基于证据的推断`、`个人分析` 区分认知状态；
- 提供一张 Mermaid 架构图和一个标记为 `说明性场景` 的任务流；
- 将固定版本、源码路径、精确 symbol 和重复来源放入主题明确的 evidence card；
- 将安全、费用、授权、恢复、版本、幂等与外部副作用边界保留在可见叙事中；
- 不虚构客户、事故、指标、基准、生产经验或上游保证。
