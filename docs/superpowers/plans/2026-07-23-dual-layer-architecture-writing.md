# Dual-Layer Architecture Case Writing Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create and validate a project-local architecture-case writing Skill, then use it to rewrite all fifteen case studies into a readable narrative layer plus an optional evidence layer without losing verified facts.

**Architecture:** Store the reusable writing workflow under `.codex/skills/writing-architecture-cases/`. Keep editorial judgment in the Skill and references, while a deterministic Node.js script reports density signals without blocking the build. Use semantic `<details>` evidence cards styled by existing global CSS, preserve the current content schema, and rewrite the articles in calibrated batches before a final cross-case review.

**Tech Stack:** Docusaurus 3.10.2, MDX 3, Node.js 24, Node test runner, CSS, project-local Codex Skills.

## Global Constraints

- Default reader: an engineer with software experience who is encountering the analyzed architecture for the first time.
- Keep every existing case slug, front matter field, official source, pinned version/commit, ten required H2 headings, and required migration H3 headings.
- Do not invent users, incidents, benchmarks, production experience, first-person experience, or upstream guarantees.
- Keep conclusion-changing limitations, security risks, permission risks, financial risks, and irreversible side-effect risks in the visible narrative.
- Use native MDX/HTML `<details>` and `<summary>` for evidence cards; add no runtime dependency.
- Treat readability thresholds as editorial warnings, not CI failures.
- A low-density passage must establish a scene, explain a term, bridge a transition, replay a task flow, pose a check, or summarize a decision.
- Commit changes in reviewable units and run fresh verification before claiming completion.

### Task 3: Add the Non-Blocking Density Reporter

**Files:**
- Create: `.codex/skills/writing-architecture-cases/scripts/analyze_case_density.mjs`
- Create: `tests/case-writing-density.test.mjs`
- Modify: `package.json`

**Interfaces:**
- Produces:

```ts
export function analyzeCaseText(
  source: string,
  options?: {
    sentenceLimit?: number;
    paragraphLimit?: number;
    consecutiveDenseLimit?: number;
  },
): {
  paragraphs: number;
  warnings: Array<{
    kind: 'long-sentence' | 'long-paragraph' | 'dense-run' | 'empty-evidence-card';
    line: number;
    message: string;
  }>;
}
```

- CLI accepts one MDX file or a directory and always exits `0` when analysis completes, even when warnings exist. Missing paths and read errors exit non-zero.

- [ ] **Step 1: Write failing analyzer tests**

Create fixtures inline in `tests/case-writing-density.test.mjs`:

```js
test('reports long prose without counting front matter, tables, or code', () => {
  const source = `---
title: Fixture
---

这是一个超过阈值的正文句子，因为它连续承担机制、版本、异常、恢复策略与迁移边界，所以应当被报告。

| 列 | 值 |
| --- | --- |
| 很长的表格内容 | 不应计入正文 |

\`\`\`ts
const longIdentifier = 'code is excluded';
\`\`\`
`;

  const result = analyzeCaseText(source, {sentenceLimit: 20});
  assert.equal(result.warnings.filter(({kind}) => kind === 'long-sentence').length, 1);
});

test('reports empty evidence cards', () => {
  const result = analyzeCaseText(
    '<details className="evidence-card"><summary>证据</summary></details>',
  );
  assert.equal(result.warnings[0].kind, 'empty-evidence-card');
});
```

- [ ] **Step 2: Run the targeted test and verify failure**

Run:

```bash

```

Expected: FAIL because the analyzer module does not exist.

- [ ] **Step 3: Implement the minimal parser**

Implement a dependency-free line scanner that:

- removes YAML front matter;

- records each paragraph’s starting line;
- checks Chinese/full-stop sentence boundaries;
- detects consecutive long paragraphs;
- detects empty `<details className="evidence-card">` blocks;
- formats CLI output as `file:line [kind] message`.
